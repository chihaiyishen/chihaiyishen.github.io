Android 项目构建相关面试问题

android studio 目录结构面试问题

在讲解 android studio 目录结构之前，我们来分清楚下 project 和 moudle 模块。如图所示 android studio 最外层就是我们的工作区间，下面的 project 会把项目相关的放在同一个区间内，这点和我们以前用 eclipse 是不一样的。eclipse 它每一个 project 是独立的，而在 android studio 里每一个 project 是可以相关连的。在每一个 project 下面，又包含了每个 module 模块。

在这里 app 这个模块

每一个模块 module 都是独立可运行的程序，这样我们就可以把若干个模块关联到 project 下面。

整个目录结构大致分为三个部分

  • 编译系统

    android studio 是用 gradle 来处理的

  • 配置文件

    配置文件会配置一些 id 文件，属性之内。

  • 应用模块

我们先来看下最外层的 Project 结构

  • .gradle

    这里包含一些 gradle 编译的脚本，gradle 是 Google 推荐的编译脚本，它的底层语音是 Groovy 的语言。gradle 编译系统版本是由我们 warpper 来指定的，因为 gradle 更新的比较快，为了兼容以前的版本，它做了一个包装 raple 这个包装类。

  • .idea

    这个文件包含了 android studio 这个工具所需要的文件，里面有很多文件夹：copyright（版权的声明），dictionaries（词典），libraries（jar 库的使用），以及配置文件。

  • app

    这里是一个 module 文件，里面是我们编写的独立业务代码。

  • build

    build 文件是指我们的 project 或者 module，编译好之后所生成的一个文件，它就会生成在 build 里面，它会生成一些 apk，jar 文件，还有一些测写的临时文件。

  • gradle

    它里面有个 warpper 文件，这是 Android 对我们的 gradle 进行的一个封装：gradle-wrapper。这样做是因为，gradle-wrapper 更新速度的非常快，它比 android studio 的更新速度快很多。所以为了兼容的角度，Google 推出了一个 gradle-wrapper.jar 的文件，然后它会更新以前旧版本的 gradle。

  • .gignore

    这文件是和我们的 git 工具有关系的，它可以配置 git 上传时不需要上传的文件。

  • build.gradle

    这文件里面是做一些相关配置的文件

  • gradle.properties

    这里就是做一些配置文件，但这里配置的是全局的，它适用于整个 module。

  • gradlew

    Linux 下可执行的文件

  • gradlew.bat

    windows 下 gradlew 的可执行文件

  • local.properties

    这是设置本地的属性，这文件不推荐上传到 git 库中。

  • settings.gradle

    这是和设置相关的，比如 gradle 的脚本会在这里设置。

在来看下 module 目录结构

  • build

    这 build 文件也是生成编译后文件存在的位置，这个文件存在的位置主要是针对这我们的 module app，也包括会生成的 apk 和临时文件也会在这里。

  • libs

    libs 主要存在第三方的 jar 文件和 aar 包，

  • src

    src 里面主要包括 java 代码和布局文件等等，我们的开发操作基本上是在这里进行的。

  • build.gradle

    整个模块相关的配置都会在这里

  • proguadrd-rules.pro

    这个文件是有个混淆配置的文件，混淆对我们的 app 的保户，缩小 app 的大小有很大的关系。

Android 构建

1.Android 构建流程

Android 把 java 文件编译成 class 字节码文件，然后把字节码文件和我们依赖的第三方的 jar 文件，打包成 classes.dex Android 可执行的文件，然后在打包我们的资源文件，在把我们的 dex 文件和 assets文件合并成未签名的包，最后通过签名打包成完整的包。

从图中可以看出，首先它有个叫 aapt 的工具。aapt 是 Android 资源打包工具，它就会将包中的 Application Resources 打包成资源文件，包括我们的 AndroidManifest.xml、layout 布局文件等等，将这些 xml 文件编译成二进制的形式。当然我们知道在 Android 当中 assets 资源文件它不会被编译，因为它图中一些图片它要保存原来的状态，不过这些资源文件它会生成资源 ID。aapt 生成之后就会加载相应的 R.java 文件。这是 Android 构建的第一步：通过 aapt 打包项目中资源文件。第二步有个 aidl 工具，这是一个 bundle 应用，这里我们会将响应 aidl 的接口转换成 Java 接口。第三步我们把所有的 R.java、Application Source Code、Jva Interfaces 通过 Java Compiler(Java 编译器) 转换成 .class 字节码文件。第四步我们会通过在 Android 当中的 dex 工具，结合上诉产生的字节码文件，打包编译成 dex 文件，dex 文件 是 Android 虚拟机中可以执行的模式。最终我们所有的 dex 文件，会打包成 apk 文件。在打包成 dex 文件之后，Android 中的 apkbuilder 会将我们编译过的资源和未编译过的资源包括图片等等，结合上面打包成的 .dex 文件，打包成我们最熟悉的 apk 文件，这就是我们的 Android 中 Package (apk)。但是这个生成的 apk 如果你要把它安装到设备上，你必须要对它签名。这就是我们的第六步：Jarsigner，这个工具对我们的 apk 进行 Debug or Release Keystore进行签名。最后当我们对 apk 进行签名之后，还需要通过 zipalign 这个工具对 apk 进行对齐操作，这样做的目的减少我们 apk 所占内存中的开销。

2.jenkins 持续集成构建

在项目开发过程中，随着我们业务需求的引进工程复杂度会增加，这时候我们不可能每开发一个版本就去打包，所以自动化的构建我们的 apk 是我们的努力的目标，而 jenkins 提供我们这样的功能。jenkins 它可以持续的集成我们的 apk，同时需要配置一些 gradle 的文件。

git

一.git 容易混淆的两个概念

1.工作区

工作区就是我们电脑当中所看见的文件目录，比如说我们的 project 就是我们的工作区。在这工作区目录中我们有个隐藏目录：.git，这里面是一些 git 版本库的内容。

2.gitigonre 文件

这文件配置一些我们不想上传到版本控制的一些文件，在这里我们可以进行配置。

二.一些常用 git 命令

1.git init

git init：创建仓库命令，有经验的开发人员，会知道调用这个命令之后，目录下会多一个 .git 的隐藏目录，这目录里面就是放置 git 版本。

2.git status

git status：查看当前目录的状态

3.git diff

git diff + 文件名：就可与看到这一次和上一次相比，我们修改了哪些内容，会已标注的形式展现出来。

4.git add

git add + 文件名：会把我们现在所要添加的文件，放到我们的栈存区当中。注意是放在栈存区当中，而不是代码区中。

5.git commit

git commit：把 git add 到栈存区的内容，提交到我们的代码区中。

我们在创建 git 版本库的时候，git 自动为我们了分支：master 分支。所以我们这里不指定分支的话，git commit 命令就会直接往 master 分支上提交命令。所以我们可以这样理解，git add 命令就是把我们所有的提交放存到我们的栈存 stack 中，然后通过 git commit 命令，把我们栈存区的所有代码，提交到我们的代码分支上。

6.git clone

git clone：从远程的仓库去克隆一份代码到本地仓库

7.git branch

git branch：查看当中的分支到底是什么

8.git checkout

git checkout：切换分支命令

三.git 的两种工作流

1.fork / clone

首先如果已 fork / clone 这种工作形式，它第一步比如说从 Joe’s GitHub repo 的远程仓库，这里可以理解成代码的远程仓库，已 fork 的形式复制一份代码仓库到我们自己的远程仓库上。复制好仓库并不是在我们的本地，而是保存在我们的远程仓库里面。这时候你的计算机本地没有对仓库做任何操作，这只是第一步。第二步 clone 你的仓库，这会让你的代码远程仓库发送一份复制文件到你的本地文件上，这时远程仓库的代码就会复制一份到你本地文件上。第三步你可以 Update a file，即更新某些文件。第四你可以通过 add 之后 commit 你的文件，然后这时候 commit 操作其实是提交到自己本地仓库上。然后可以通过第五步的 push 在把本地仓库中的代码提交到远程的仓库上，这就完成了第五步。第六步：pull request，即你要远程仓库中的代码提交到项目中的仓库上，这时候你需要发送一个 pull request 命令。项目仓库的管理员会看到你发送的 pull request，他会核实你的代码。如果你的代码没有太大的错误，他就会把你代码合并到仓库上。

2.clone

clone 形式就是跳过 fork 到自己的远程仓库中，直接 clone 到本地仓库中，然后就 Push 到项目仓库，这样就缺少了管理员。而 fork / clone 它有一个管理员，它可以保证代码的质量。

gradle 面试问题讲解

在我们创建一个新的项目的时候会创建三个新的 gradle 文件，一个是 project 下面的 build.gradle 和 settings.gradle，以及我们的 app 包里面的 build.gradle。

settings.gradle

settings.gradle 文件它只包含在一个 Android 的项目当中，对于一个新的项目来说，它只会包含一个 moudle 模块，这个文件就是包含 moudle 模块用的。settings.gradle 文件它在初始化阶段会被执行，它会定义 app new 的 moudle 在我们的构建范围内。如果是一个人单独开发的话，你不一定需要这个文件。但如果是多模块开发的话，每个人开发不同的模块，这时候你需要 settings.gradle 来编制一定的规则，让我们的业务逻辑开发更有条理。

bulid.gradle(project)

这文件一般包含两个代码块：buildecript、allprojects，实际的构建它会在 buildecript 这个代码块里

buildscript {
    repositories {
        jcenter()
    }
    dependencies {
        classpath 'com.android.tools.build:gradle:2.3.0'

        // NOTE: Do not place your application dependencies here; they belong
        // in the individual module build.gradle files
    }
}

repositories 是我们代码的仓库，意味着依赖包，比如我们可以依赖第三方的 jar 包或者代码，这里 jcenter() 是一个很有名的代码库。

dependecies 是代码块在构建过程中所需要依赖的那些包，这就意味你不能将你的应用或者依赖项目所需要的依赖包，包含在顶层的构建文件当中，因为默认的情况下是 Android 的 gradler 应用插件。每一个 Android 模块都需要有 Android 插件，因为该插件可以执行该模块的相关任务。

allprojects {
    repositories {
        jcenter()
    }
}

allprojects 是用来定义模块的属性，你甚至也可以在这里创建任务，在上面我们可以看到它也有 repositories 代码块。但需要注意的是你一旦使用 allprojects 属性，这个模块就耦合到你所在的项目当中。这就意味着很有可能在主项目没有构建的情况下，你这个模块是无法构建的。所以说这在刚开始 Android studio 这是一个问题，你后面如果想把它单独的分离到自己的项目当中，你就需要重新创建 builde 文件。

build.gradle(app)

apply plugin: 'com.android.application'

android {
    compileSdkVersion 25
    buildToolsVersion "25.0.2"
    defaultConfig {
        applicationId "cn.com.mjj.coolspider.myapplication"
        minSdkVersion 15
        targetSdkVersion 25
        versionCode 1
        versionName "1.0"
        testInstrumentationRunner "android.support.test.runner.AndroidJUnitRunner"
    }
    buildTypes {
        release {
            minifyEnabled false
            proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
        }
    }
}

dependencies {
    compile fileTree(dir: 'libs', include: ['*.jar'])
    androidTestCompile('com.android.support.test.espresso:espresso-core:2.2.2', {
        exclude group: 'com.android.support', module: 'support-annotations'
    })
    compile 'com.android.support:appcompat-v7:25.3.1'
    compile 'com.android.support.constraint:constraint-layout:1.0.1'
    testCompile 'junit:junit:4.12'
}

这个文件是属于 app 自己的模块，而且它是可以覆盖顶层的 build.gradle 文件。这文件一般有 android 和 dependencies 两个标签所构成，dependencies 主要用于我们模块所依赖的一些包。

在上面的配置文件中，我们看第一行就是我们所用到的 Android 插件。在顶层配置构建文件当中，这个插件被配置成我们一些依赖。这是由谷歌的工具团队复杂 Android 插件编写和维护的，所以这一块是官方提供的，它用于构建、测试、打包 Android 项目所使用。

接下来看下 android 的标签，在这里包含全部的 Android 所有的配置。这些配置之所以可以使用，是因为第一行配置了 Android 插件。其中有很多属性，但有两个属性是必备的：compileSdkVersion、buildToolsVersion。compileSdkVersion 表示用来编译应用的 Android api 版本号，buildToolsVersion 表示用于构建和编译使用的版本号。这里还有一个 defaultConfig，这里就是整个代码配置的核心属性，我们来依依看下。

  • applicationId

    这属性覆盖了 AndroidMainfest.xml 里的 package name，即你可以用它来覆盖我们的包的名字。不过这个 applicationId 和包名它有一些不同，在 gradle 被作用默认 Android 构建系统之前，这 AndroidMainfest.xml 和 packageName 它两个用途：作为应用的唯一标记和在 as 资源中它使用的包名。

  • minSdkVersion

    配置运行最小的 Android api 版本

  • targetSdkVersion

通知应用已经在某个 Android 版本已经通过测试，就可以让系统不用在向前兼容。

  • versionCode

    版本号

  • versionName

    版本名称，构建文件中的属性都可以覆盖 Mainfest.xml 中的属性。

接下来看下 dependencies 依赖包，依赖包它是我们标准配置的一部分，这就是把它放在 android 代码块之外的原因。虽然它是标准的，但是它是独立于 Android 标准的一部分。在这里我定义了应用和所有项目的依赖包，所有的依赖包都定义在这里。

Proguard 代码混淆面试问题

一. Proguard 到底是什么

ProGuard 工具是用于压缩，优化，混淆我们的代码，主作用是可以移除代码中的无用类，字段，方法和属性同时可以混淆。最终会使我们打包后的 apk 体积更加小，也可以让我们的 apk 更加难以编辑。

二.ProGuard 技术的功能

1.压缩

在打包的时候，通过 ProGuard 来检查并移除代码中没有用到的类、方法和属性。因为我们在业务迭代的时候，有时候会用到别人的开发过的一些业务模块。有时如果你把它代码删掉了，即没用引用它的时候，这段代码它其实是没有用的。但如果你把它删掉的话又怕以后有用，这时候你用 ProGuard 技术，你可以在打包的时候把无用的类删掉就不会影响日常开发。

2.优化

ProGuard 它会优化通过 Java-C 编译后的字节码文件，它会移除那些无用的 .class 文件里的指令。

3.混淆

如果我们使用一些无意义的名称来命名类，例如：1234，abcd 这些名称，这样我们的类的字段方法可能重名，这时候即使我们的代码被编译了对方也很难读懂。这就是 ProGuard 的作用，它会把开发当中有意义的名称变成无意义的名称，这样恶意的竞争对手反编译了我们的 apk 仍然是无法知道我们代码是做什么用的。

4.预检测

在 Java 平台对那些处理过后的代码在进行检测

三.ProGuard 工作原理

EntryPoint

在讲 ProGuard 工作原理之前，我们要先引入 EntryPoint 概念。EntryPoint 我们可以把它理解成一个标志，它是在 ProGuard 处理过程中不会被处理的类或者方法。在 PorGuard 进行压缩过程的时候，它会从上述的 EntryPoint 类中，开始搜索那些类和它的成员在使用。被标记为 EntryPoint 的类和方法，有些在使用。并且这些在配置混淆配置文件中，不希望被混淆的类和方法，但是有些它是没有使用的。对于那些没有使用的类和成员，在压缩阶段被我们的 PorGuard 所丢弃。这在进一步优化 ProGuard 中，它不是 EntryPoint 的类，我们就会把它设置为不是参数也会被移除。最后 PorGuard 会对那些没使用 EntryPoint 的类，进行重名，以便于我们预检测来保证我们代码的稳定性。

为什么要进行混淆

我们知道 Java 是跨平台的解释性语言，而 Java 的源代码会被编译成字节码保存在我们 .class 文件中。由于跨平台的需要，Java 字节码中包括了很多源代码的信息，变量名、方法名等等，并且通过这些名称来访问变量和方法。这些符号其实很多是无用的，但是又很容易被反编译成源代码。为了防止这个现象，我们就需要通过 ProGuard，来对 Java 的字节码进行混淆。混淆就是对发布出去的程序，进行重新的组织和处理。ProGuard 的作用就是对那些即将发布出去的程序，进行重新的组在这种和处理。使得处理后的代码和未处理过的代码，有相同的功能，但是代码完全不一样，同时这个代码不容易被反编译。而且即使你真的反编译成功之后，也很难读懂这代码是做什么用的。那些被混淆过的代码，仍然遵循原来的格式去执行，执行结果是一样的。所以说这就是 ProGuard 的作用，它对外保证了我们程序的安全性，但是对内是透明的，程序执行结果是一样的。
